Technical Design Rationale

Alarma

Adaptive Anti-Oversleep Destination Alarm & Emergency Safety System — the reasoning behind every technology and architecture decision.
Project: Capstone & Application Development
Program: BS Information Technology — Polytechnic University of the Philippines
Development Team: BS Information Technology Capstone Project Group

Document: Design & Technology Justification

1Purpose of This Document

This companion to the README answers a single question for every major decision in Alarma: why this, and not the alternative? It documents the reasoning behind the platform, the architecture, the data and security choices, and the production build — so reviewers can evaluate the engineering judgement, not just the result.

Guiding principles. Three constraints shaped nearly every choice below: the app must work offline-first (commuters lose signal underground and on long routes), it must be privacy-first by law (Republic Act No. 10173, the Data Privacy Act of 2012), and it must remain diagnosable in the field where no debugger is attached.

2Platform & Framework

Why .NET MAUI on .NET 9

Chosen: .NET MAUI

Why: A single C# codebase compiles to a native Android app with direct access to platform APIs (foreground services, SmsManager, Keystore) that the safety features depend on. The team's existing C#/.NET fluency meant no new language ramp-up, and .NET 9 brings mature AOT and IL-trimming used for the hardened Release build.

Over the alternatives: Flutter/React Native would have meant a new language and a plugin bridge for every native security call; native Kotlin would have delivered one platform at a higher cost with no benefit, since iOS is explicitly out of scope.

Why Android-only (API 26+)

Chosen: net9.0-android

Why: The target users — Metro Manila jeepney, UV Express, and city-bus commuters — are overwhelmingly on budget and mid-range Android devices. API 26 (Android 8.0) as the floor covers the vast majority of in-use handsets while still giving access to modern foreground-service and notification APIs. iOS, web, and desktop are deliberately out of scope to keep the safety surface small and well-tested.

Why the MVC architecture

Chosen: Model–View–Controller

Why: A clear separation between data models, XAML views, and controllers keeps the security-critical logic (validation, crypto, location handling) in testable, UI-independent classes. That separation is what makes the standalone xUnit test suite possible without any Android/MAUI dependency.

3Data Storage & Persistence

Why SQLCipher over plain SQLite

Chosen: sqlite-net-pcl + SQLCipher (AES-256)

Why: All structured data — trip history, saved routes, emergency contacts, geocode cache, behavioral profiles — is sensitive. SQLCipher encrypts the entire database file at rest with AES-256, so a lost, stolen, or rooted device yields nothing readable. sqlite-net-pcl gives a clean ORM with parameterized queries only, eliminating SQL-injection surface.

Over the alternatives: Plain SQLite stores data in cleartext on disk; a flat JSON/Preferences store offers neither encryption nor query integrity.

Why split Preferences vs. SecureStorage

Chosen: Preferences for settings, Keystore for secrets

Why: Non-sensitive UI settings live in fast MAUI Preferences; every cryptographic key lives in the Android Keystore via SecureStorage. No key is ever written to Preferences or hardcoded in the binary, so decompiling the APK reveals no secrets.

One key, one home. The database, backup, and black-box keys are each 256-bit, generated by RandomNumberGenerator, and stored only in the Keystore (slots alarma_db_key_v1, alarma_backup_key_v2, alarma_blackbox_key_v2). This single rule — random keys, Keystore-resident, never hardcoded — underpins the entire cryptographic story.

4Security & Cryptography

Security was a primary design constraint, not an afterthought. The threat model assumes an attacker with physical or filesystem access to the device.

Why AES-256-GCM everywhere encryption is local

Chosen: AES-256-GCM (AesGcm)

Why: GCM is authenticated encryption — it provides confidentiality and integrity in one pass. Any tampering with the ciphertext, nonce, or tag makes decryption fail before a single byte is returned. Both the encrypted backup and the forensic black-box log verify the GCM tag before any JSON is deserialized, closing the door on data-integrity attacks (OWASP A08).

Over the alternatives: AES-CBC would require a separate HMAC step and is easy to get wrong; using the platform's System.Security.Cryptography avoids hand-rolled crypto entirely.

Why a Forensic Black Box Logger

Chosen: encrypted, Keystore-backed crash log

Why: Faults happen in the field — on a moving jeepney at night — where no debugger or ADB exists, and a Release build emits no Debug.WriteLine output. The logger seals crashes with AES-256-GCM, then on next launch decrypts the prior fault to a readable report and deletes the source (consumed once). The key is preloaded at startup so the crash handler can encrypt synchronously without awaiting, and the writer never throws — there is no crash-on-crash loop.

Why validate-before-clear on restore

Chosen: defensive restore design

Why: A naïve restore that wipes the database first could destroy live user data if the backup is corrupt. Alarma validates every record — length caps, Philippine phone-number format, coordinate bounds, quantity caps — before touching the existing database. A tampered or empty backup therefore cannot silently erase a user's contacts or routes (OWASP A04).

5Location, Mapping & Geocoding

Why an Android foreground service for tracking

Chosen: foreground Service, Exported=false

Why: The destination alarm must keep computing distance even when the screen is off and the app is backgrounded — exactly what Android kills a normal background task for. A foreground Service with ForegroundServiceType=TypeLocation (required from API 34) keeps tracking alive; Exported=false means no other app can start or bind it. Coordinates are held in-memory as immutable values and never logged or persisted.

Why a PARTIAL_WAKE_LOCK

Chosen: scoped wake lock

Why: Aggressive OEM battery managers (common on the budget phones our users carry) throttle GPS delivery when the CPU sleeps, which would make the alarm miss the stop. A PARTIAL_WAKE_LOCK is held only while location providers are registered and released on stop — keeping delivery reliable without leaking the lock or draining the battery.

Why Leaflet + CartoDB in a WebView, not a native map SDK

Chosen: Leaflet.js, no API key

Why: Google Maps SDK requires an API key, a billing account, and sends usage telemetry — at odds with the no-backend, privacy-first design. Leaflet with CartoDB dark tiles renders in a MAUI WebView under a strict Content-Security-Policy, needs no API key and no personal identifiers, and the dark theme suits late-night use.

Why Photon primary + Nominatim fallback + local aliases

Chosen: layered open geocoding

Why: Destination search is the only feature that needs the network, so it uses free, open services with no keys. Photon is fast and typo-tolerant; Nominatim is the fallback when Photon misses; and a client-side Philippine alias dictionary resolves local landmark names (e.g., colloquial stop names) that neither service indexes well. Geocode results are cached in the encrypted database to cut repeat lookups.

6Emergency SOS

Why cellular SMS, not an internet push

Chosen: Android SmsManager

Why: An emergency is exactly when there is no Wi-Fi and no mobile data — but cellular SMS still works. Sending live coordinates directly through Android SmsManager means SOS functions fully offline and never passes through any Alarma server, so there is no backend to fail, leak, or subpoena.

Safeguards: a 2-second press-and-hold prevents accidental triggers (timer cleared on page exit), and a 30-second cooldown prevents runaway sending. The action gives a short haptic + beep confirmation but deliberately does not hijack the ringer or Do-Not-Disturb (see Section 7).

7Capstone Audit Resolution

A focused functional and UX audit ahead of the panel defense closed twelve defects in the commute pipeline and the safety features. None of these changes alter the cryptographic story in Sections 3–4 — the SQLCipher database, the AES-256-GCM backup and black box, and the Keystore key handling are all untouched.

Infrastructure & permissions

Mandatory GPS enforcement

Pre-flight the device location switch

Why: A user can grant the app location permission yet still have the phone's master location switch off — in which case no fix ever arrives and a "trip" would track nothing. Starting a trip now checks ILocationService.IsLocationServiceEnabled() first and, if location is off, blocks the start and sends the user straight to the OS location settings — the same way a navigation app refuses to guide without GPS.

Two-way permission toggles

Honor the OFF path

Why: The setup switches only ever requested permissions; flipping one off did nothing. Android offers no API to revoke a runtime permission programmatically, so OFF now tears down local state (an active tracking service and its wake lock are stopped) and routes the user to App Settings, where the OS actually allows revocation. On return, the switch re-syncs to the real status.

Alarm, tracking & state

Haversine distance with a GPS-jitter gate

Noise-gated path integral

Why: Trip distance is summed as a true Haversine path integral between fixes. A stationary phone still emits fixes that wander a few metres, which would inflate the total, so a per-segment jitter gate discards any movement below the fix's own accuracy (minimum 8 m) and only advances the comparison anchor on an accepted segment — rejecting noise while still accumulating genuine slow movement.

Full state reset on trip stop

No ghost state

Why: Stopping a trip used to leave the destination set, so a stale View Active Trip affordance lingered on the home screen. Stop now performs a complete reset — destination and map marker cleared, search header restored, the in-memory active trip dropped, and the foreground service killed — returning the home screen to its idle state.

Monotonic, outlier-resistant escalation

Persisted arrival

Why: The three alarm stages are monotonic by construction (lead ≥ 300 m > arrival 200 m > overshoot), and arrival must now hold across consecutive fixes before it latches — so a single outlier fix can no longer fake an arrival, nor the false overshoot that would chase it. The vehicle-type selector was removed in favor of a single adaptive, speed-based lead distance.

Feedback & SOS

SMS-first SOS with a discreet cue

No ringer / DND hijack

Why: SOS previously fired the full multi-stage alarm and overrode Do-Not-Disturb — the opposite of what someone in a vulnerable situation needs. It is now strictly an SMS dispatch with a short haptic buzz and a brief confirmation beep so the press registers quietly. The cue respects the phone's current volume and DND, so a silenced phone stays silent.

Smaller UX fixes in the same pass. Tapping a trip notification now deep-links straight to the active-trip screen via an explicit PendingIntent through a SingleTop activity; the origin row reads "Searching for GPS…" instead of "Location unavailable" while a fix is acquired; and the emergency-contact form shows inline feedback for both validation errors and successful saves instead of failing silently.

Post-audit refinements

Frictionless search: the destination becomes a tappable field

Reuse the search screen, don't inline an editor

Why: Once a destination was picked, its text went read-only and the only way to change it was the small back chevron — easy to miss. We restyled it as a light-lavender pill with an edit pencil (matching the search bars) and made the whole pill tap straight back into the focused search screen. We deliberately reused the existing search page instead of swapping in an inline Entry: that keeps all the search logic in one place and, just as important, never clears the destination — so the hardened stop-trip state resets stay intact. The initial and active search bars were unified to the same #EDE7F6 lavender so every search surface reads as one consistent control.

Real-time location in the SOS message

Active-trip fix → last-known → text-only fallback

Why: An SOS is only useful if a contact knows where to go, so the SMS now appends a tappable https://maps.google.com/?q={lat},{lon} link. At the instant the button is pressed we take the freshest fix through a fallback chain — the active trip's in-memory Haversine coordinate first, then the platform's last-known/fresh location, and if neither is available the plain SOS text still sends so contacts are alerted regardless. Every step is a passive read, so grabbing the coordinate can't disturb the tracking loop the alarm depends on.

UX & Data Portability

Why we moved backups from hidden AppData to native OS folder selection

Chosen: FileSaver "Save As" + FilePicker import

Why: The first cut of Backup/Restore wrote the encrypted .alarma file into the app's private AppData directory and read the "latest" one back automatically. It worked, but it quietly took ownership of the user's data away from them — a commuter could never see the file, copy it to another phone, hand it to a classmate, or store it in their own Drive. For an offline, privacy-first app whose entire promise is that the user's data belongs to the user, hiding the one portable artifact of that data in storage they cannot reach was the wrong default. We therefore route export through the native OS "Save As" dialog (Community Toolkit FileSaver) so the rider picks the destination folder themselves — typically Downloads — and route import through the native file browser (FilePicker) so they navigate to and select that exact file back. The cryptographic envelope is unchanged (AES-256-GCM, Keystore key, validate-before-clear restore); only the file's location moves out of hidden storage into a folder the commuter genuinely controls.

Over the alternatives: Auto-writing to a fixed public path (e.g. a hardcoded Downloads location) would have re-introduced scoped-storage permission friction on modern Android and still denied the user a real choice; the system pickers grant one-shot access to exactly the file or folder the user points at, with no broad storage permission required.

Non-Intrusive Feedback

Why transient "grey pill" notifications replaced blocking modals

Chosen: auto-hiding toast over DisplayAlert

Why: Low-stakes confirmations — "backup saved", "backup restored" — were originally shown with DisplayAlert, a modal that dims the screen and demands a tap before anything else can happen. On a moving jeepney at night, that is exactly the wrong interaction: it seizes the whole screen and the user's attention for a message they only need to glance at, right when their attention belongs on their surroundings and their stop. We replaced those modals with a transient "grey pill" — a rounded, dark, toast/snackbar-style capsule that fades in near the bottom of the screen, reports the outcome, and auto-dismisses after a few seconds without ever blocking input. Feedback the rider can absorb at a glance and ignore, instead of an interruption they must clear. We deliberately keep true modals only where a decision genuinely must be made and acknowledged (destructive deletes, invalid-contact errors, the location-off prompt); everything informational degrades to the non-blocking pill.

8Interface, Iconography & Layout Engineering

The UI is judged on a moving vehicle, one-handed, often at night. These decisions are about legibility, predictable touch targets, and assets that scale cleanly across the wide range of screen densities our commuter devices ship with.

Why Material Symbols font ligatures over raw emojis

Chosen: MaterialSymbolsOutlined.ttf

Why: The interface originally leaned on raw system emojis for glyphs like search, location, and the favorite star. Those render differently on every OEM skin and Android version — wrong weight, wrong color, sometimes a low-fidelity bitmap — so the UI looked inconsistent across the exact budget handsets our users carry. We migrated every icon to native Google Material Symbols font ligatures, bundled as MaterialSymbolsOutlined.ttf and registered once in MauiProgram.cs under the MaterialSymbols font alias. A glyph is now just text (Text="location_on"), so it inherits font size and color, stays vector-sharp at any density, and looks identical on every device.

Over the alternatives: Per-icon PNGs would multiply asset count across density buckets and still rasterize; emojis surrender all control of weight and color to the platform.

Why the onboarding became a single asset-mapped image per slide

Chosen: full-design PNG + transparent tap zones

Why: The tutorial went through a few layouts — an aspect-fit box that letterboxed the art, then a layered Grid stack, then a CarouselView. Each design ran the title, subtitle and buttons as live MAUI controls drawn over the illustration, and every one had its own occlusion or input bug. The carousel was the worst offender: on Android its RecyclerView item containers are clickable, so the system click sound fired no matter where on the screen the rider tapped, not just on the buttons. We settled on the simplest design that can't go wrong: each slide is now a single full-bleed PNG (onboarding_full_1…4.png) with the title, subtitle, progress dots and the Skip / Next pills already baked into the artwork, mapped one-to-one from the reference mockups. The image is shown Aspect="AspectFill" with no gesture recognizer, so it silently absorbs every stray tap — no carousel, no screen-wide click sound. Only two transparent Grid tap zones are laid over the baked-in Skip and Next pills, so the click sound and the navigation now fire only on the buttons.

Updated navigation logic: Tapping Next swaps the source to the next PNG; Skip jumps straight to the last slide. On the final slide both actions raise a modal consent popup carrying the Privacy Policy and Terms & Conditions links plus an agreement checkbox — Get Started stays inert until it is ticked. Only then does onboarding persist HasSeenTutorial and HasAgreedToTerms and route to //home. HomeView keeps the gate: if either flag is unset it forwards back to onboarding, and once both are set it advances to the permissions-setup screen — so the legal consent can never be skipped.

Over the alternatives: Keeping the carousel meant fighting the native click sound on every container; rebuilding the title/subtitle/dots as live controls re-invited the occlusion and alignment bugs that earlier layouts had. Baking the chrome into the image and swapping a single source is both pixel-faithful to the mockups and impossible to mis-hit-test.

Why a standardized gold-yellow favorite accent

Chosen: Material Gold-Yellow #FFC107

Why: The favorite star in the saved-location list rendered in the same muted purple as the rest of the theme, so it read as decoration rather than a state the user could recognize and act on. Standardizing the star on Material Design gold-yellow (#FFC107), applied directly in the list item template so it colors every entry, gives a high-contrast, universally understood "favorited" signal that survives the dark theme and dim, glare-prone commute lighting.

9Production Build Hardening

The Release configuration does double duty: it shrinks and speeds up the APK and it raises the cost of reverse-engineering.

SettingWhy we enabled it
RunAOTCompilationCompiles managed IL to native code, so there is no IL left to decompile in the shipped APK.
AndroidEnableProfiledAotProfile-guided AOT prioritizes hot paths (alarm trigger, GPS callback, geocoding) for faster cold start.
PublishTrimmed / TrimMode=fullRemoves unreferenced code across the whole dependency closure — smaller binary, less dead-code attack surface.
AndroidLinkMode=SdkOnlyStrips unused SDK/AndroidX code at the DEX level.
DebugSymbols=falseNo symbols or sequence points, so decompilers cannot map native code back to source lines.
Why TrimmerRoots.xml exists. The IL trimmer cannot see types created by reflection. Because the SQLite ORM instantiates model classes reflectively, the AlarmaApp.Models namespace and the SQLite assemblies are explicitly rooted so full trimming does not delete classes that are only ever referenced at runtime.

10Testing Strategy

Why a separate net9.0 xUnit project

Chosen: AlarmaApp.Tests (plain net9.0)

Why: The security-critical logic — phone-number validation, coordinate bounding, alarm-sound whitelisting, lead-time clamping, backup restore caps, AES-GCM length checks, and the speed-based multi-stage alarm — must be verifiable without spinning up an Android emulator. Targeting plain net9.0 with no MAUI dependency keeps the suite fast and runnable in CI, which is only possible because MVC kept that logic out of the views.
